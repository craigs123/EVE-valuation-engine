Technical Brief: ESA CCI Land Cover Data API Access Implementation
Overview
The application accesses ESA CCI (Climate Change Initiative) Land Cover data through OpenLandMap's STAC (SpatioTemporal Asset Catalog) API, not through direct ESA endpoints. The implementation reads Cloud-Optimized GeoTIFFs (COGs) to extract pixel values at specific coordinates.

ESA Land Cover Data Specification
Collection Identifier: "land.cover_esacci.lc.l4"
Data Source: ESA CCI Land Cover L4 Global Maps
Resolution: Multiple resolutions available (typically 300m global)
Format: Cloud-Optimized GeoTIFF
Value Type: Categorical integer codes representing land cover classes

API Access Implementation
1. STAC Catalog Integration
Base URL: https://s3.eu-central-1.wasabisys.com/stac/openlandmap

Land Cover Code Processing
Raw Value Extraction
Extracts integer class codes directly from GeoTIFF pixels
No scaling or transformation applied to raw values
Out-of-bounds coordinates return null
Code Translation Map
const landCoverTypes: Record<number, string> = {
  10: 'Cropland',
  50: 'Tree Cover', 
  130: 'Grassland',
  220: 'Snow/Ice'
};
const coverType = landCoverTypes[result.value] || `Class ${result.value}`;
Output Format
Category: "landcover"
Unit: "class"
Value: Human-readable string (e.g., "Tree Cover")
Raw Code: Original integer preserved in processing pipeline

2. Result Processing
Raw pixel values categorized as landcover
Integer codes converted to descriptive strings
Unknown codes displayed as "Class {code_number}"
Assigned Font Awesome icon: "fas fa-map"
3. Error Handling
Bounds Checking: Validates coordinates within GeoTIFF extent
Asset Validation: Ensures valid GeoTIFF asset exists
Network Resilience: Graceful failure if collection unavailable
No Synthetic Fallbacks: Returns null if real data unavailable
Technical Specifications
Dependencies
geotiff: JavaScript library for reading GeoTIFF files
fetch: For STAC API calls
TypeScript: Strong typing for data structures
Performance Considerations
Direct COG Access: Reads only required pixel region, not entire file
Caching Opportunity: STAC metadata could be cached (not currently implemented)
Network Overhead: Each query requires 3 HTTP requests (collection → item → asset)
Response Structure
{
  collection: "land.cover_esacci.lc.l4",
  name: "Land Cover", 
  category: "landcover",
  value: "Tree Cover", // Translated from integer code
  unit: "class",
  metadata: {
    title: string,
    description: string,
    extent: object,
    license: string,
    assetUrl: string
  }
}
Integration Points
Frontend Display
Results appear in "Land Cover & Vegetation" section
Shows descriptive text rather than raw codes
Includes metadata tooltip with technical details
Schema Validation
Input coordinates validated via coordinateQuerySchema
Output structured according to EnvironmentalData type
Limitations & Considerations
Limited Code Dictionary: Only 4 common land cover types have human-readable names
No Temporal Selection: Always uses most recent available data
Single Pixel Sampling: No spatial aggregation or smoothing
No Confidence Metrics: ESA quality flags not currently extracted
This implementation prioritizes authentic ESA data delivery over convenience features, ensuring users receive genuine land cover classifications without synthetic interpolation.
